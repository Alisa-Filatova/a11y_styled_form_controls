Pattern Details

Pattern Markup 
<fieldset class="fieldset">
  <legend>Group Title</legend>

  <div class="r-group">
    <input type="radio" id="r1" name="radios" checked>
    <label for="r1">Option 1</label>

    <input type="radio" id="r2" name="radios">
    <label for="r2">Option 2</label>
  </div>
</fieldset>

When customizing any form control, it must remain accessible to assistive technology and keyboard users. To keep this custom style accessible, a series of visually hidden radio button form controls are included in the pattern. They allow keyboard and screen reader users to discover the control and select an option.

The radio buttons are wrapped in a fieldset element. The presence of the fieldset element groups the radio buttons and programmatically communicates that they are functionally interdependent. The legend element provides a description of the grouped radio buttons' purpose.

If it makes sense for the grouping of radio buttons to be called out in the document outline, then the legend can incorporate a heading element.

The input with the checked attribute applied to it will trigger a CSS selector that visually updates the appearance of the sibling label associated with it, indicating that it is the selected option.

A selector targeting the last label's :after pseudo-element will create an outline around the entire component to indicate it has keyboard focus and can be interacted with.

Note:

Radio buttons are meant to indicate a group of single choice options, typically part of a larger form area of a document. It is considered a WCAG 3.2.2, On Input violation to change state of an interface when interacting with an input that is not announced as a "button" (e.g. a button or input with a type value of button, reset or submit).

This pattern may be perfectly acceptable to use in the context of a form or as a UI updating mechanism, so long as it is accompanied by an a "confirmation" or "submit" button.

If a "confirmation" or "submit" button is not present, an alternative implementation of this pattern may be more appropriate. Consider using ARIA attributes such as aria-pressed, which will communicate current state on a grouping of interrelated button elements. This is commonly used on applications where state is immediately updated after being activated.
